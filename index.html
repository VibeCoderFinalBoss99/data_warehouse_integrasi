Perilaku API & Validasi Input

Seluruh endpoint menerapkan validasi data yang ketat untuk memastikan konsistensi relasi database data warehouse.

Sistem Penamaan Kolom & Alias JSON Struktur database asli memiliki kolom yang mengandung spasi atau karakter khusus (seperti L/P, ID SEKOLAH). Untuk mempermudah integrasi frontend, API secara otomatis memetakan key tersebut menjadi format JSON camelCase/snake_case standar.
Validasi Existence Check Sebelum melakukan operasi PUT (Update) atau DELETE (Hapus), API akan melakukan query SELECT terlebih dahulu untuk memverifikasi keberadaan data. Jika data tidak ada, API akan mengembalikan HTTP 404.

Aturan Validasi Berdasarkan Metode

Metode Validasi Wajib Perilaku Gagal
POST Field utama data tidak boleh null, undefined, atau string kosong. Mengembalikan HTTP 400 (Bad Request)
PUT Field id wajib diisi dan merupakan integer positif. Field data yang diperbarui tidak boleh kosong. Mengembalikan HTTP 400 jika format salah, atau HTTP 404 jika ID tidak ditemukan.
DELETE Parameter id wajib diisi (melalui request body atau query parameter) dan harus berupa integer positif. Mengembalikan HTTP 400 jika format salah, atau HTTP 404 jika ID tidak ditemukan.

Panduan Endpoint CRUD (Contoh Endpoint: /dim_sekolah)

Seluruh endpoint dimensi memiliki pola pemanggilan dan format payload yang seragam. Berikut adalah contoh request dan response-nya.

GET /dim_sekolah?limit=2&offset=0
Mendapatkan daftar asal sekolah yang terpaginasi. Endpoint mendukung parameter query limit dan offset.

Response (200 OK)

{
  "status": "success",
  "message": "Data dim_sekolah berhasil diambil",
  "meta": {
    "limit": 2,
    "offset": 0,
    "count": 2
  },
  "data": [
    {
      "nama_sekolah": "SMAN 1 BANDUNG",
      "ID SEKOLAH": 1999
    },
    {
      "nama_sekolah": "SMAN 2 BANDUNG",
      "ID SEKOLAH": 1998
    }
  ]
}

Headers Kehadiran Cache

X-Cache: HIT / MISS
Cache-Control: no-store

Header X-Cache: HIT menandakan payload dibaca langsung dari cache memori serverless untuk kecepatan transmisi data.

POST /dim_sekolah
Menyimpan data asal sekolah baru ke dalam database. Nilai ID auto-generated akan dikembalikan di response.

Request Body (JSON)

{
  "nama_sekolah": "SMAN 3 BANDUNG"
}

Response (201 Created)

{
  "status": "success",
  "message": "Data berhasil ditambahkan",
  "data": {
    "id": 2000,
    "nama_sekolah": "SMAN 3 BANDUNG"
  }
}
PUT /dim_sekolah
Mengubah data sekolah berdasarkan key id. Operasi ini dilindungi verifikasi existence check.

Request Body (JSON)

{
  "id": 2000,
  "nama_sekolah": "SMAN 3 BANDUNG REVISI"
}

Response (200 OK)

{
  "status": "success",
  "message": "Data berhasil diperbarui",
  "data": {
    "id": 2000,
    "nama_sekolah": "SMAN 3 BANDUNG REVISI"
  }
}
DELETE /dim_sekolah?id=2000
Menghapus data sekolah dari database. Parameter ID bisa dikirim via request body {"id": 2000} atau query string.

Request URL / Parameter

DELETE /dim_sekolah?id=2000

Response (200 OK)

{
  "status": "success",
  "message": "Data berhasil dihapus"
}

HTTP Status & Struktur Error Response

API menggunakan standard status code HTTP untuk menginformasikan status pemrosesan instruksi:

Code Status Skenario
200 OK Sukses GET, PUT, DELETE
201 Created Sukses POST
400 Bad Request Input kosong, ID invalid, JSON rusak
404 Not Found ID tidak ditemukan di database
405 Not Allowed Method HTTP tidak disupport
500 Server Error Masalah koneksi database / server

Contoh Response Validasi Gagal (HTTP 400)

{
  "status": "error",
  "message": "Field 'nama_sekolah' wajib diisi dan tidak boleh kosong"
}

Contoh Response Data Tidak Ada (HTTP 404)

{
  "status": "error",
  "message": "Data dengan ID 9999999 tidak ditemukan"
}
